Modular, Composable, Distributed, Cascading Microauthorizations and “Action Spaces” in Cryptographic Ledgers

I’ve been thinking a bit as I implement a POC in Openchain. REST can be implemented in WebSockets. That’s what Dual API makes possible: it brings an HTTP-like protocol to WebSocket connections using a hierarchical event emitter.

Irrespective to the technology, REST is built-to-last: “REpresentational State Transfer, or REST, is a set of constraints that ensure a scalable, fault-tolerant and easily extendible system.”

I’ve actually been writing code to demonstrate an implementation of Openchain in the dual protocol context, such that the entire API can be made REST-ful, rather than just the /stream endpoint.

Ultimately this can turn all requests for Asset Definitions, ACC records, DATA records, etc into submillisecond requests that can be cache invalidated across hierarchically generated permissions such that we could practically implement “cascaded permissions”.

Where asset public key paths collocate permissioned actions such that broadcasts of the ledger can be distributed across keys within socket connections BEFORE hitting the Validator Node. What this means is that fault tolerance can be built into HTTP-like system within the socket BEFORE the transaction stream of the Validator node itself is even requested to post transactions.

Post Transactions can be distributed within the protocol messaging layer itself. This is “decentralize the Web.” So instead of distributing the database itself — the protocol to interact with Openchain would fundamentally be distributed within WebSocket connections. Such that the transaction ledger is only updated via a system of distributed clients within hierarchically determined domain channels which are mapped to assets (which represent permissions) in Openchain — this is what I mean by “modular, composable microauthorizations”.

A use-case is distributed validations across hierarchically determined domains which cascade permissions for localizable, functional validations, rather than purely centralized validations. Since Openchain is arguably more decentralized than Bitcoin, because organizations implement their own server instances (though Openchain assumes (1) you’ll implement your own client, whereas my project is to facilitate an elegant server-to-server Openchain network, and (1.1) that you are comfortable with client-side crypto — in JavaScript, and (1.2) offers no “persistence wallet” via LocalStorage, so write down your mnemonic!), the ideas expressed here suggest a distributed, potentially Byzantine-fault tolerant way to enable self-organizing “validation oracles” at the protocol layer for connecting clients. This way, Openchain can have Observer nodes which keep the Validator node in check, while “channels” at the protocol layer can be built to scale subsystems failure of collocated transaction stream updates before central validation. Thus, discoverability and extensibility can be built into evolutionary API controls of the distributed system. Adding to this, cascading, localizable, dynamic fault-tolerance mechanisms can be implemented (or embedded) to extend discovery implementation details of (JSONAPI) payloads.

Ultimately, these ideas should culminate into robust schema extensions to the Action Spaces of Schema.Org. Even further, algorithmic complexity of distributed messaging over WebSockets can be managed and measured to align the ledger-as-a-service’s “fault lines” or communicative texture by automated, heuristic models leveraging the “science of imaginary solutions” (‘Pataphysics) where a “creative encounter” engine can live within the distributed system to detect said system faults and dispatch signalling for dynamic self-repair of subsystems and back-propagation of discoverability constraints. Connecting clients contribute as weights to the oracle’s real-time confirmations of global transactional integrity and ACID-evaluations.
